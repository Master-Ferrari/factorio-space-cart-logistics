# Space Cart Logistics — ТЗ (v2.3)

## Цель
Factorio-мод (2.0 / Space Age): тайл-размерные рельсы и каретки, едущие по ним как «смесь конвейера и поезда». **Это не пасфайнд**: у кареток нет маршрута A→B, они просто текут. «Программа» движения живёт в состоянии соединений тайлов (задаётся сигналами), интеллект в сети, не в каретке. Вся механика на Lua; родные trains/rails/cars/LuaRendering **не используются**.

## Стек
Factorio 2.0 API (Lua 5.2). `storage`, не `global`. Файлы: `info.json`, `data.lua`, `control.lua`, `graphics/`.

## Нейминг
- Mod: **Space Cart Logistics**. Internal name (`info.json` `name`, путь `__space-cart-logistics__`): `space-cart-logistics`. Автор: `gofarovich`.
- **Префикс всех именованных регистраций — `gofarovich-scl-`**: прототипы, item'ы, entity, рецепты, custom-input'ы, remote-интерфейсы, шорткаты и т.п. Напр.: `gofarovich-scl-rail`, `gofarovich-scl-cart`.
- Lua-идентификаторы (где дефис недопустим — ключи/функции/локали) — под тем же неймспейсом, но через подчёркивание: `gofarovich_scl_`.

## Жёсткие решения (не менять)
- **Рельс = две сущности на одном тайле** (пересмотр v2.3; причина — `simple-entity-with-owner` не подключается к проводам, а комбинатор не умеет `graphics_variation`):
  - **Примари** `gofarovich-scl-rail` = `constant-combinator` (клон). То, что игрок **ставит/майнит/выбирает**, к чему **цепляются провода и GUI**. Спрайт прозрачный (`sprites = empty`). 1×1, `collision_mask = {layers={}}`. Единственная выбираемая сущность на тайле → провод всегда целится в неё. Это `node.entity` в storage; нативно читает/выводит цепь (отдельный компаньон не нужен).
  - **Арт** `gofarovich-scl-rail-art` = `simple-entity-with-owner`, `selectable_in_game = false`, без коллизии. Несёт `pictures`(64) + `graphics_variation` (комбинатор так не умеет). Скрипт создаёт/держит/красит её по маске и сносит вместе с примари. Это `node.art` в storage; узел графа + спрайт.
- **Каретка** = `simple-entity-with-owner` (НЕ `car`/`unit`), 1×1, нулевая коллизия. Реальный entity → родной свет/тень.
  - `flags` включают **`placeable-off-grid`** — иначе `teleport` снаппит на центр тайла. Критично.
  - Позиция: только `teleport`, дробные координаты. `speed` **не задавать никогда** (ноль физики).
  - Поворот: через `graphics_variation` (**не** `direction`). `pictures` = лист 16 кадров, `random_variation_on_create = false`; в runtime `graphics_variation = facing` (1..16).
  - Груз: данные в `storage`. Id: `unit_number`.
- Стиль: data-oriented, главный цикл наш (`on_tick`). Всё детерминированно (мультиплеер): фиксированный порядок обхода и тай-брейки.
- **Инвариант:** каждая каретка всегда принадлежит ровно одному составу (размер ≥ 1). Соло-каретка = состав из одного, тот же код-путь.

## Тайл рельса
До **6 ненаправленных соединений** между сторонами N/S/E/W (все пары): прямые N–S, E–W; повороты N–E, N–W, S–E, S–W. Состояние соединений задаётся сигнальными условиями (см. Сигналы); по умолчанию — из конфига.
**Снапшот на входе:** каждая каретка фиксирует состояние соединений тайла в тик, когда её **голова входит в тайл**, и держит решение до конца проезда. Смена условий не трогает уже вошедшую — применяется к следующей. Именно это **расщепляет составы**.
**Выбор выхода** при входе со стороны `in` (кандидаты — где соединение включено и `out ≠ in`, разворота нет):
`прямо → направо → налево → стоп`. Направо = по часовой относительно курса: N→E, E→S, S→W, W→N. Прямо всегда в приоритете; поворот — только если прямо закрыто.

## Сигналы
- У каждого из 6 соединений — своё **условие** (сигнал/оператор/константа, как у combinator). Истинно → соединение включено. Условие может меняться в любой тик.
- **Чтение цепи:** решено через двусущностную модель рельса (см. «Жёсткие решения») — примари сам = `constant-combinator`, к нему цепляются провода, сигналы читаем/выводим из его сети через Lua. (Проверено: `simple-entity-with-owner` коннектор в 2.0 НЕ поддерживает; отдельный скрытый компаньон отвергнут — конфликт выбора курсора с рельсом.)
- 6 условий > 1 нативного → условия это **наши данные** на тайл (×6), задаются кастомным GUI, вычисляются в Lua. Для UPS считать **по событию смены сигналов**, не каждый тик на каждый тайл.
- Результат: 6 булей → битмаска соединений → драйвит и рисование (`graphics_variation`), и маршрутизацию.
- **Все 6 ложны (битмаска 0):** тайл не рисуем (пустой кадр) — и это **немедленно** (override отложенного применения): каретки на его клетках **визуально взрываются** (спавн взрыва + удаление каретки + чистка состава). Частичная смена откладывается по снапшоту; полный блэкаут убивает.

## Интерфейс тайла (GUI, M6)
Открывается кликом по примари-рельсу (`on_gui_opened`). Окно из двух панелей. Референс макета строки условия и шапки «connected to» — соседний проект `factorio-button-combinator` (`add_condition_row`, `add_connection_status`; `COMPARATORS = {"<", ">", "=", "≥", "≤", "≠"}`, правый операнд = одиночный слот с тумблером константа/сигнал).

**Левая панель «Cart rail»:** титул + крестик; **вьюпорт** — картинка тайла с текущими активными направлениями; снизу чекбоксы **manual** и **circuit network**.
- **По умолчанию (auto):** направления выводятся из соседей автоматически (как сейчас). Виден только чекбокс `manual`; чекбокса `circuit network` нет, правой панели нет. Вьюпорт — просто read-only картинка.
- **manual ON:** во вьюпорте появляются галочки (правка путей) **и** появляется чекбокс `circuit network`. Галочек **всегда 6** — игрок волен включить любой из 6 путей, даже без соседнего рельса (путь «в никуда» просто стопнёт каретку на краю тайла). Ручная маска полностью переопределяет авто.
- **circuit network ON:** раскрывается правая панель.

**Правая панель «connected to …»:** список **строк-условий** — по строке на каждый **активированный** путь (число строк = число включённых галочек, не всегда 6). Каждая строка: цветной индикатор пути + условие как у комбинатора `[сигнал] [оператор] [сигнал | константа]`. Снизу — чекбокс **read next cart content**.
- **Семантика:** база доступности пути = auto(соседи) или manual(ручная маска). Если circuit ON: путь ВКЛ ⇔ *база* **AND** *условие истинно*. **Пустое (не заданное) условие = считать отсутствующим → путь активен всегда.**

**read next cart content** (зависит от груза кареток — пока НЕ реализуем, в ТЗ ещё не описано как предметы попадают в каретки):
- «content» каретки = её содержимое (склоняемся к **3 стака** на каретку).
- «следующая каретка» = та, чья **голова войдёт в этот тайл в следующем тике**.
- влияние: payload следующей каретки **мерджится** к сигналам сети при оценке тех же 6 условий.
- вывод: payload следующей каретки **выводится в провода** тайла на **1 тик**.
- груз пока задаём debug-командой (станции загрузки — M7/позже).

**Цвета 6 соединений** (общие для арта вьюпорта и индикаторов строк): `N-S=red, E-W=cyan, N-E=purple, N-W=yellow, S-E=green, S-W=orange`.

**Вьюпорт-галочки:** точное наложение на дуги в Factorio GUI почти невозможно (нет абсолютного позиционирования) → для моков размещаем приблизительно по сетке **3×3** (компас-якоря), поправим позже.

**storage (добавления к рельсу/каретке):**
- `storage.rails[key] += { mode="auto"|"manual", manual_mask, circuit(bool), conditions={ [conn]={signal,comparator,use_signal,second_signal,constant} }, read_next(bool), eff_mask }` — `eff_mask` итог, драйвит `graphics_variation` арта + маршрут.
- `storage.carts[un] += { payload }`.

**Под-этапы M6** (порядок реализации):
- **6a.** GUI-каркас (read-only): открытие, титул+close, вьюпорт-картинка, чекбоксы manual/circuit без логики справа. Локали ru/en. Модуль `scripts/gui.lua`.
- **6b.** Manual-маска: 6 галочек, ручной `eff_mask`, auto↔manual, маршрут+`graphics_variation` уважают `eff_mask`. (Блэкаут-взрыв при mask 0 — **отложен**, см. ниже.)
- **6c.** Модель условий + правая панель: 6 строк по образцу, хранение `conditions`, чекбокс circuit, строки только для базово-доступных путей, пустое условие = всегда истинно. Оценка «как данные» (без реальной сети).
- **6d.** Чтение сети: примари-комбинатор читает red+green; пересчёт 6 булей **по событию смены сигналов** (троттлинг, не каждый тик); `eff_mask = base ∧ conditions`.
- **6e.** Cart payload (отложено): модель груза + debug-команда.
- **6f.** read next cart content (отложено): чтение груза следующей каретки → в оценку условий + 1-тиковый вывод.
- **Отложено в рамках M6:** блэкаут-взрыв (п.5 обсуждения — «пока не трогаем»), payload и read-next (нужна модель груза/загрузки из M7).

## Рисование рельса
Финальный арт — **один лист 64 ячейки** (8×8, 64×64 px/ячейка, 512×512; `scale = 0.5` → 1 тайл). Рисуется вручную, чтобы корректно сводить стыки/кресты (over/under, единые узлы — не простым наложением). Мод-сторона: `pictures` = этот лист, `variation_count = 64`, `line_length = 8`; в runtime `rail.graphics_variation = mask + 1` (вариации 1-индексные — проверить 0/1). Арт доходит до краёв ячейки (стыковка соседей). «Настоящие» тени — отдельным таким же 64-листом на shadow-слое. Всё — часть entity, без LuaRendering. Для хруста на близком зуме можно 128 px/ячейку → лист 1024×1024, `scale = 0.25`.

**Контракт «бит → ячейка»** (раз и навсегда, общий для арта и кода):
- Биты: `0=N–S, 1=E–W, 2=N–E, 3=N–W, 4=S–E, 5=S–W`. `mask` = OR активных (0..63).
- Ячейка = `mask` row-major, 8 в ряд: `col = mask & 7`, `row = mask >> 3`. mask 0 — верх-лево, 63 — низ-право.
- **Ячейка 0 (mask 0) = полностью прозрачная** = невидимый тайл (все соединения off).

**Моки (сейчас):** арт делает сам агент программно. Для каждой `mask` рисует активные сегменты (прямые — линиями, повороты — дугами r=0.5) в 64×64 и складывает в 8×8 по тому же контракту; наложением — для моков сойдёт. Аналогично мок каретки: 16 кадров-стрелок. Реальный арт автор подставит позже, контракт не меняется. **На первых порах сидим на моках.**

## Геометрия и дискретность
- Скорость = жёлтая лента = 1/32 тайла/тик. **1 клетка = 1 тик ≈ 1/32 тайла.**
- Прямой тайл = **32 клетки**. Поворот (четверть окружности r=0.5, длина π/4) = 8π ≈ 25.13 → **25 клеток** (вниз).
- Размер клетки ≈ одинаков везде (разница 0.5%) → длина и дистанция однородны в клеточном пространстве.
- Каретка едет **ровно 1 клетка/тик**. Старт/стоп мгновенные (без ускорения).
- Каретка = **скользящее окно ~32 клетки** (физ. длина). На повороте занимает 2–3 тайла (напр. 2+25+5=32). Обязательно, иначе состав визуально сжимается на углах.

## Движение составов (snake-deque)
- Состав = непрерывный ряд кареток бампер-в-бампер = **дек занятых клеток** (голова спереди, хвост сзади). Границы кареток = оффсеты в деке; спрайт каретки — в её центральной клетке, поворот по касательной → `graphics_variation`.
- **Движение** (ехать/стоять): гейтит голова состава; голова встала → весь состав встал. Сдвиг push/pop — на весь состав разом.
- **Маршрут** (какой выход): у **каждой** каретки свой, по снапшоту соединений на входе в тайл (`прямо→право→лево→стоп`). Пока соседние каретки выбирают тот же выход — состав цел; тайл сменился между их входами и каретка выбрала иначе, чем передняя → **раскол** на два состава.
- Тело каждой каретки ретрейсит **её собственный** сохранённый след (не пересчитываем) → корректно на развязках и при смене соединений на лету.
- **Join/split** — события: догнал впритык и тот же маршрут → влился; зазор или расхождение маршрута → раскол.
- **Кольцо**: если цепочка `ahead` замкнулась (головы нет) и ни одной клетке не мешает чужой → всё кольцо проворачивается на 1 клетку/тик атомарно. Полное кольцо без пустой клетки всё равно крутится. Стопорит только внешняя занятость.

## Коллизии и арбитраж
- Оккупанси **по клеткам**; два состава конфликтуют при пересечении деков.
- Спорная клетка перекрёстка остаётся в деке состава, пока его хвост её не покинет → встречный ждёт **весь состав** (выпадает само).
- Две независимые головы на одну свободную клетку в один тик:
  1. ПДД — уступи правому (даёт параллелизм, никто не голодает).
  2. Тай-брейк для симметрии/циклов: абсолют **N > E > S > W**.
- **Вне скоупа пока:** кросс-составной gridlock (два состава держат клетки друг друга) — отдельная задача (детект цикла ожиданий / резерв перекрёстка).

## storage (эскиз, агент уточнит)
- `storage.rails[pos_key] = { pos, conns, conditions }` — `conns` = текущая битмаска, `conditions` = 6 условий.
- `storage.cells` — оккупанси по клеткам (клетка → состав), для O(1) проверок.
- `storage.convoys[id] = { deque клеток, каретки с оффсетами }`.
- `storage.carts[unit_number] = { entity, convoy_id, offset, payload }`.

## Цикл тика
1. (по событию/троттлингу) пересчитать условия тайлов → битмаска → `graphics_variation`; обработать блэкаут (взрыв занявших клетки кареток).
2. Для каждого состава посчитать желаемую клетку головы.
3. Разрешить конфликты на спорных клетках (ПДД → N>E>S>W).
4. push/pop у едущих; обновить позиции и `graphics_variation` кареток.
5. Пересобрать join/split (с учётом расхождения маршрутов) и кольца.

## Этапы
1. **База (готово):** прототипы рельса/каретки (off-grid, graphics_variation, teleport), граф рельс, одна каретка едет по связному пути.
2. **Клеточная модель:** path→клетки (32/25), каретка = окно 32 клетки, 1 клетка/тик, поворот по касательной. Одна каретка по правилу `прямо→право→лево→стоп`. Проверка: нет сжатия на повороте.
3. **Один состав:** snake-deque, push/pop, движение от головы, маршрут по снапшоту, join/split на линии прямая+поворот. Голова встала → все встали.
4. **Кольцо:** детект цикла + атомарный проворот; полное кольцо без пузырька крутится.
5. **Перекрёстки + арбитраж:** клеточные конфликты, ожидание всего состава, ПДД → N>E>S>W. Две пересекающиеся струи.
6. **Сигналы:** circuit-компаньон/connector, чтение сети, 6 условий + GUI, битмаска → `graphics_variation` (64 вариации) + маршрут по снапшоту (расщепление составов), блэкаут-взрыв.
7. **Позже:** кросс-составной gridlock; станции загрузки/выгрузки (реальный entity с инвентарём, мост к лентам/инсёртерам; payload каретки — данные).
