# Space Cart Logistics — ТЗ (v2.4)

> **v2.4 (пересмотр модели сигналов):** условия стали **направленными** и **списочными**.
> Геометрия пути задаётся только галочками (как в v2.3); куда поедет каретка — решает
> упорядоченный список условий вида «(вход → выход): предикат» (12 видов = 4 входа × 3 поворота).
> Старая модель «6 условий по соединению, гейтящих геометрию» отменена.

## Цель
Factorio-мод (2.0 / Space Age): тайл-размерные рельсы и каретки, едущие по ним как «смесь конвейера и поезда». **Это не пасфайнд**: у кареток нет маршрута A→B, они просто текут. «Программа» движения живёт в состоянии соединений тайлов (задаётся сигналами), интеллект в сети, не в каретке. Вся механика на Lua; родные trains/rails/cars/LuaRendering **не используются**.

## Стек
Factorio 2.0 API (Lua 5.2). `storage`, не `global`. Файлы: `info.json`, `data.lua`, `control.lua`, `graphics/`.

## Нейминг
- Mod: **Space Cart Logistics**. Internal name (`info.json` `name`, путь `__space-cart-logistics__`): `space-cart-logistics`. Автор: `gofarovich`.
- **Префикс всех именованных регистраций — `gofarovich-scl-`**: прототипы, item'ы, entity, рецепты, custom-input'ы, remote-интерфейсы, шорткаты и т.п. Напр.: `gofarovich-scl-rail`, `gofarovich-scl-cart`.
- Lua-идентификаторы (где дефис недопустим — ключи/функции/локали) — под тем же неймспейсом, но через подчёркивание: `gofarovich_scl_`.

## Жёсткие решения (не менять)
- **Рельс = две сущности на одном тайле** (пересмотр v2.3; причина — `simple-entity-with-owner` не подключается к проводам, а комбинатор не умеет `graphics_variation`):
  - **Примари** `gofarovich-scl-rail` = `constant-combinator` (клон). То, что игрок **ставит/майнит/выбирает**, к чему **цепляются провода и GUI**. Спрайт прозрачный (`sprites = empty`). 1×1, `collision_mask = {layers={}}`. Единственная выбираемая сущность на тайле → провод всегда целится в неё. Это `node.entity` в storage; нативно читает/выводит цепь (отдельный компаньон не нужен).
  - **Арт** `gofarovich-scl-rail-art` = `simple-entity-with-owner`, `selectable_in_game = false`, без коллизии. Несёт `pictures`(64) + `graphics_variation` (комбинатор так не умеет). Скрипт создаёт/держит/красит её по маске и сносит вместе с примари. Это `node.art` в storage; узел графа + спрайт.
- **Каретка** = `simple-entity-with-owner` (НЕ `car`/`unit`), 1×1, нулевая коллизия. Реальный entity → родной свет/тень.
  - `flags` включают **`placeable-off-grid`** — иначе `teleport` снаппит на центр тайла. Критично.
  - Позиция: только `teleport`, дробные координаты. `speed` **не задавать никогда** (ноль физики).
  - Поворот: через `graphics_variation` (**не** `direction`). `pictures` = лист 16 кадров, `random_variation_on_create = false`; в runtime `graphics_variation = facing` (1..16).
  - Груз: данные в `storage`. Id: `unit_number`.
- Стиль: data-oriented, главный цикл наш (`on_tick`). Всё детерминированно (мультиплеер): фиксированный порядок обхода и тай-брейки.
- **Инвариант:** каждая каретка всегда принадлежит ровно одному составу (размер ≥ 1). Соло-каретка = состав из одного, тот же код-путь.

## Тайл рельса
До **6 ненаправленных соединений** между сторонами N/S/E/W (все пары): прямые N–S, E–W; повороты N–E, N–W, S–E, S–W. **Геометрия** (какие пути существуют и рисуются) = `eff_mask`: auto (из присутствующих соседей) или manual (галочки вручную). **Условия геометрию НЕ меняют** (пересмотр v2.4 — раньше circuit гейтил маску); они влияют только на выбор выхода.
**Снапшот на входе:** каждая каретка фиксирует состояние соединений тайла в тик, когда её **голова входит в тайл**, и держит решение до конца проезда. Смена условий не трогает уже вошедшую — применяется к следующей. Именно это **расщепляет составы**.
**Выбор выхода** при входе со стороны `in`: сначала — **направленные условия** входа `in` (см. Сигналы): первое истинное сверху задаёт выход. Если условий нет / ни одно не истинно (частый случай) — **дефолтное правило** по включённым путям (`out ≠ in`, разворота нет): `прямо → направо → налево → стоп`. Направо = по часовой относительно курса: N→E, E→S, S→W, W→N. Прямо всегда в приоритете; поворот — только если прямо закрыто.

## Сигналы и условия (направленная модель, v2.4)
- **Геометрия ≠ маршрут.** Какие пути есть на тайле — только галочки (`manual_mask`) или авто-соседи; это драйвит арт (`graphics_variation`) и набор *возможных* выходов. **Условия геометрию НЕ гейтят** — они выбирают, куда из доступных выходов поедет каретка.
- **Условие = направленное правило** «(вход → выход): предикат». Видов условий — **12** (4 входа × 3 поворота, разворота нет; ≡ 6 соединений × 2 направления проезда). Количество условий **не ограничено**, **дубли разрешены**.
- **Группировка по входу:** условия одного входа образуют категорию (Top/Right/Bottom/Left in). Категория появляется ⇔ есть ≥1 её условие. Порядок категорий не важен; **порядок внутри категории = приоритет** (выше = выигрывает для следующей каретки).
- **Маршрут на входе `in`:** идём по списку `cond_lists[in]` сверху вниз; **первое** условие, чей предикат **истинен** И чей выход — **включённый путь** (`CONN[in][exit]` есть в `eff_mask`), задаёт выход. Иначе — дефолтное правило (см. «Тайл рельса»). Снапшот на входе по-прежнему расщепляет составы.
- **Предикат** = `[сигнал] [оператор] [сигнал | константа]` (как у комбинатора; `COMPARATORS = {"<", ">", "=", "≥", "≤", "≠"}`, правый операнд = один слот с тумблером константа/сигнал). **Пустой предикат (сигнал не выбран) = всегда истинно** → безусловный выход (catch-all). У условия есть необязательное **имя** — косметика, ни на что не влияет.
- **Чтение цепи:** примари-рельс сам = `constant-combinator` (см. «Жёсткие решения»), к нему цепляются провода; red+green читаем/мерджим в Lua. Считаем **по событию смены сигналов** (троттлинг), не каждый тик на каждый тайл. (Проверено в v2.3: `simple-entity-with-owner` коннектор в 2.0 НЕ поддерживает; скрытый компаньон отвергнут — конфликт выбора курсора.)
- **Устаревшие условия:** игрок снял галочку пути → условия с этим выходом показываем **disabled** и в маршруте игнорируем (данные не теряем).
- **Блэкаут** (`eff_mask == 0`, все галочки сняты): тайл невидим, каретки на его клетках **взрываются** (спавн взрыва + удаление + чистка состава). Привязан к **маске (геометрии)**, НЕ к условиям. *(Отложено в M6.)*

## Интерфейс тайла (GUI, M6)
Открывается кликом по примари-рельсу (`on_gui_opened`, подавляем нативный combinator-GUI). **Одно окно** (тащится нативно целиком; «приклеить» два top-level окна Factorio не умеет, ручное репозиционирование через `on_gui_location_changed` лагает на кадр — отвергнуто): сверху панель «Cart rail», **снизу** — панель условий.

**Левая панель «Cart rail»:** титул + крестик; **вьюпорт** — картинка тайла с активными путями (стопка GUI-слоёв с альфой: база + цветной путь на каждый бит `eff_mask`; **не камера** — она клампит зум ~3x); снизу чекбоксы **manual** и **conditions** (бывш. «circuit network»).
- **auto (по умолчанию):** пути выводятся из соседей. Виден только `manual`, правой панели нет, вьюпорт read-only.
- **manual ON:** во вьюпорте 6 галочек правки путей (всегда 6 — путь «в никуда» просто стопнёт каретку на краю тайла) **и** появляется чекбокс `conditions`. Ручная маска переопределяет авто.
- **conditions ON:** снизу в том же окне разворачивается панель условий (мастер-переключатель `conditions_on`).
- **Важно (v2.4):** галочки = ТОЛЬКО геометрия (что нарисовано и какие выходы возможны), драйвят `graphics_variation`. Куда поедет каретка — решают условия (или дефолтное правило).

**Панель условий (снизу, без заголовка; тёмный контейнер `inside_deep_frame` = фон между категориями) — динамический список (НЕ 6 фикс-строк):**
- Со старта **пуст**. Снизу — кнопка **«New condition»** → поп-ап **«Select direction»**: сетка 5×5 из 12 иконок (вход → выход), 3 на вход; кнопки, чей путь выключен галочкой, **disabled**. Клик закрывает поп-ап и добавляет **пустое** условие выбранного вида в категорию его входа.
- Условия **группируются по входу** в категории «<side> in» (Top/Right/Bottom/Left). Порядок категорий на маршрут **не влияет** (хранится в `node.cat_order` чисто для отображения, реордер ↑/↓ на шапке); **порядок внутри категории = приоритет** (выше = выигрывает). Дубли разрешены, число не ограничено.
- **Плоский список светлых карточек на тёмном контейнере (`inside_deep_frame`):** заголовок категории (`bordered_frame`: ↑/↓ + стрелка-входа ▼▲◀▶ + «<side> in» + ✕) и под ним условия той же категории — карточки `bordered_frame` **с отступом слева**. Крестики категории и условия одного размера.
- **Строка условия:** `[↑/↓ слева, стопкой]` · `[иконка вход→выход]` · …растяжка… · `[сигнал]` · `[оператор]` · `[операнд]` · `[тумблер константа/сигнал]` · `[✕]`. (Имя условия пока убрано.) У **категории** ✕ удаляет все её условия.
- **Операнд-константа** = квадратная кнопка-слот с числом, сокращённым до ~3 символов (43k, 1.7M); клик открывает поп-ап ввода (ползунок + поле + «Set constant»). Сигнал-операнд = обычный `choose-elem-button`. Тумблер переключает между ними.
- **Реордер:** нативного drag-to-reorder для мод-GUI в 2.0 **нет** (Rseding91, форум: «нет общего drag-and-drop списка, каждый в игре — кастомный»). → кнопки **↑/↓** (слева, стопкой).
- **Устаревшие условия:** сняли галочку пути → условия с этим выходом показываем **disabled**, в маршруте игнорируем.

**read next cart content** (зависит от груза кареток — пока НЕ реализуем, в ТЗ ещё не описано как предметы попадают в каретки):
- «content» каретки = её содержимое (склоняемся к **3 стака** на каретку).
- «следующая каретка» = та, чья **голова войдёт в этот тайл в следующем тике**.
- влияние: payload следующей каретки **мерджится** к сигналам сети при оценке условий.
- вывод: payload следующей каретки **выводится в провода** тайла на **1 тик**.
- груз пока задаём debug-командой (станции загрузки — M7/позже).

**Цвета 6 соединений** (для арта вьюпорта; индикатор строки условия теперь — иконка направления, а не цветной квадрат): `N-S=red, E-W=cyan, N-E=purple, N-W=yellow, S-E=green, S-W=orange`.

**12 иконок направлений** (поп-ап «Select direction» + индикатор строки): sprite-прототипы `gofarovich-scl-dir-<вход>-<выход>` (вход/выход ∈ N/E/S/W). Мок рисует `tools/gen_directions.ps1` → `graphics/directions/<вход><выход>.png` (64px, путь от ребра входа к ребру выхода + стрелка наружу на выходе). Одни и те же иконки в поп-апе и в строке. *(Реализовано: генератор + регистрация спрайтов; GUI-использование — впереди.)*

**Раскладка поп-апа (5×5, как галочки путей):** 12 иконок по периметру компасом — кластер входа у своего ребра, поворот-влево/прямо/поворот-вправо по позиции выхода. Кнопка disabled, если её путь (`CONN[вход][выход]`) выключен галочкой.

**Вьюпорт-галочки:** точное наложение на дуги в Factorio GUI почти невозможно (нет абсолютного позиционирования) → размещаем приблизительно по сетке **3×3** (компас-якоря), поправим позже. *Реализовано (6b):* 3×3 сетка чекбоксов кладётся поверх спрайта вьюпорта через negative-margin; раскладка — повороты по углам, прямые N-S сверху-центр / E-W слева-центр.

**storage (рельс):**
- `storage.rails[key] += { mode="auto"|"manual", auto_mask, manual_mask, conditions_on(bool), eff_mask, cond_lists, read_next(bool) }`.
  - `conditions_on` — **мастер-переключатель** (чекбокс `conditions`, доступен только в manual): гейтит И панель условий, И их применение в маршруте. Off → условия игнорируются (дефолт-роутинг).
  - `eff_mask` = базовая маска (manual переопределяет auto; **условия её НЕ гейтят** — пересмотр v2.4) → драйвит `graphics_variation` арта, `node.conns`/`node.mask` и набор возможных выходов.
  - `cond_lists = { [entry] = { {exit, name, signal, comparator, use_signal, second_signal, constant}, ... } }` — на каждый вход `entry`∈N/E/S/W упорядоченный список (приоритет сверху вниз), дубли ок; `exit` — один из 3 поворотов входа.
  - **Маршрут:** если `conditions_on` — первое условие `cond_lists[in]`, чей `CONN[in][exit]` включён в `eff_mask` (`node.conns`-гейт убирает устаревшие) И предикат истинен, задаёт выход; иначе — дефолтное правило `прямо→право→лево→стоп`.
- `storage.gui_open[player.index] = key` — какой тайл открыт у игрока (роутинг событий GUI).
- `storage.carts[un] += { payload }` — позже (модель груза M7).

## Рисование рельса
Финальный арт — **один лист 64 ячейки** (8×8, 64×64 px/ячейка, 512×512; `scale = 0.5` → 1 тайл). Рисуется вручную, чтобы корректно сводить стыки/кресты (over/under, единые узлы — не простым наложением). Мод-сторона: `pictures` = этот лист, `variation_count = 64`, `line_length = 8`; в runtime `rail_art.graphics_variation = eff_mask + 1` (вариации 1-индексные — проверить 0/1). Арт доходит до краёв ячейки (стыковка соседей). «Настоящие» тени — отдельным таким же 64-листом на shadow-слое. Всё — часть entity, без LuaRendering. Для хруста на близком зуме можно 128 px/ячейку → лист 1024×1024, `scale = 0.25`.

**Контракт «бит → ячейка»** (раз и навсегда, общий для арта и кода):
- Биты: `0=N–S, 1=E–W, 2=N–E, 3=N–W, 4=S–E, 5=S–W`. `mask` = OR активных (0..63).
- Ячейка = `mask` row-major, 8 в ряд: `col = mask & 7`, `row = mask >> 3`. mask 0 — верх-лево, 63 — низ-право.
- **Ячейка 0 (mask 0) = полностью прозрачная** = невидимый тайл (все соединения off).
- **GUI-вьюпорт (отдельная графика, НЕ этот лист):** окно собирает тайл стопкой из 7 текстур с альфой 256×256 — `gofarovich-scl-vp-base` + `gofarovich-scl-vp-<conn>` (6 цветных путей), `graphics/viewport/`, рисует `tools/gen_viewport.ps1`. Слои кладутся друг на друга negative-margin'ом и растягиваются на вьюпорт (`stretch_image_to_widget_size`); порядок наложения = порядок битов. Лист `rail.png` остаётся только для арт-сущности в мире.

**Моки (сейчас):** арт делает сам агент программно. Для каждой `mask` рисует активные сегменты (прямые — линиями, повороты — дугами r=0.5) в 64×64 и складывает в 8×8 по тому же контракту; наложением — для моков сойдёт. Аналогично мок каретки: 16 кадров-стрелок. Реальный арт автор подставит позже, контракт не меняется. **На первых порах сидим на моках.**

## Геометрия и дискретность
- Скорость = жёлтая лента = 1/32 тайла/тик. **1 клетка = 1 тик ≈ 1/32 тайла.**
- Прямой тайл = **32 клетки**. Поворот (четверть окружности r=0.5, длина π/4) = 8π ≈ 25.13 → **25 клеток** (вниз).
- Размер клетки ≈ одинаков везде (разница 0.5%) → длина и дистанция однородны в клеточном пространстве.
- Каретка едет **ровно 1 клетка/тик**. Старт/стоп мгновенные (без ускорения).
- Каретка = **скользящее окно ~32 клетки** (физ. длина). На повороте занимает 2–3 тайла (напр. 2+25+5=32). Обязательно, иначе состав визуально сжимается на углах.

## Движение составов (snake-deque)
- Состав = непрерывный ряд кареток бампер-в-бампер = **дек занятых клеток** (голова спереди, хвост сзади). Границы кареток = оффсеты в деке; спрайт каретки — в её центральной клетке, поворот по касательной → `graphics_variation`.
- **Движение** (ехать/стоять): гейтит голова состава; голова встала → весь состав встал. Сдвиг push/pop — на весь состав разом.
- **Маршрут** (какой выход): у **каждой** каретки свой, по снапшоту на входе в тайл (условия входа, иначе `прямо→право→лево→стоп`). Пока соседние каретки выбирают тот же выход — состав цел; условия/сигналы сменились между их входами и каретка выбрала иначе, чем передняя → **раскол** на два состава.
- Тело каждой каретки ретрейсит **её собственный** сохранённый след (не пересчитываем) → корректно на развязках и при смене соединений на лету.
- **Join/split** — события: догнал впритык и тот же маршрут → влился; зазор или расхождение маршрута → раскол. *Реализовано:* `do_joins`/`do_splits` пре-пассом каждый тик (предикат «`next_head` задней == хвостовая клетка передней + близкий facing», раскол — точная инверсия слияния, не осциллируют). **Раскол выполняется ДО движения** — иначе отколовшаяся каретка едет как тело состава (без проверки оккупанси) и наезжает на переднюю «фронт в фронт»; после раскола у её состава своя голова с гейтом оккупанси.
- **Кольцо**: если цепочка `ahead` замкнулась (головы нет) и ни одной клетке не мешает чужой → всё кольцо проворачивается на 1 клетку/тик атомарно. Полное кольцо без пустой клетки всё равно крутится. Стопорит только внешняя занятость.

## Коллизии и арбитраж
- Оккупанси **по клеткам**; два состава конфликтуют при пересечении деков.
- Спорная клетка перекрёстка остаётся в деке состава, пока его хвост её не покинет → встречный ждёт **весь состав** (выпадает само).
- Две независимые головы на одну свободную клетку в один тик:
  1. ПДД — уступи правому (даёт параллелизм, никто не голодает).
  2. Тай-брейк для симметрии/циклов: абсолют **N > E > S > W**.
- **Вне скоупа пока:** кросс-составной gridlock (два состава держат клетки друг друга) — отдельная задача (детект цикла ожиданий / резерв перекрёстка).

## storage (эскиз, агент уточнит)
- `storage.rails[pos_key] = { pos, conns, eff_mask, cond_lists }` — `conns`/`eff_mask` = геометрия (галочки/соседи), `cond_lists` = направленные условия по входу (см. «Интерфейс тайла»).
- `storage.cells` — оккупанси по клеткам (клетка → состав), для O(1) проверок.
- `storage.convoys[id] = { deque клеток, каретки с оффсетами }`.
- `storage.carts[unit_number] = { entity, convoy_id, offset, payload }`.

## Цикл тика
1. (по событию/троттлингу) пересчитать `eff_mask` тайлов (галочки/соседи) → `graphics_variation`; обработать блэкаут (взрыв занявших клетки кареток). Маршрут (условия входа) считается при входе каретки в тайл, не здесь.
2. Для каждого состава посчитать желаемую клетку головы.
3. Разрешить конфликты на спорных клетках (ПДД → N>E>S>W).
4. push/pop у едущих; обновить позиции и `graphics_variation` кареток.
5. Пересобрать join/split (с учётом расхождения маршрутов) и кольца.

## Этапы
1. **База (готово):** прототипы рельса/каретки (off-grid, graphics_variation, teleport), граф рельс, одна каретка едет по связному пути.
2. **Клеточная модель:** path→клетки (32/25), каретка = окно 32 клетки, 1 клетка/тик, поворот по касательной. Одна каретка по правилу `прямо→право→лево→стоп`. Проверка: нет сжатия на повороте.
3. **Один состав:** snake-deque, push/pop, движение от головы, маршрут по снапшоту, join/split на линии прямая+поворот. Голова встала → все встали.
4. **Кольцо:** детект цикла + атомарный проворот; полное кольцо без пузырька крутится.
5. **Перекрёстки + арбитраж:** клеточные конфликты, ожидание всего состава, ПДД → N>E>S>W. Две пересекающиеся струи.
6. **Сигналы** (в работе — 6a/6b/6d/6e/6f готовы, старый 6c отменён v2.4; дальше 6g троттлинг + отложенные read-next/блэкаут): GUI тайла, **направленные условия** (12 видов «вход→выход», список по входу, first-match-wins, фолбэк на дефолт), чтение сети примари-комбинатором, геометрия по галочкам → `graphics_variation` (64 вариации), блэкаут-взрыв.

**UX-правки (вне этапов):** каретка `selection_priority=60` (выбор поверх рельса); рельс под кареткой не удаляется (добыча → пересоздание с переносом проводов + flying-text; деконструкция отменяется) — `C.tile_has_carts`, `R.recreate_entity`; разворот одной каретки по R (`custom-input` → `C.reverse_cart`: откол из состава + переворот следа). Заодно фикс `facing_close` (был mod 16 при 32 кадрах).
7. **Позже:** кросс-составной gridlock; станции загрузки/выгрузки (реальный entity с инвентарём, мост к лентам/инсёртерам; payload каретки — данные).
