# Жёсткие решения (не менять)
> Часть ТЗ Space Cart Logistics — индекс в [../readme.md](../readme.md). Читать перед правкой `data.lua` / прототипов.

- **Рельс = ОДНА сущность на тайл** (пересмотр v2.5; двухсущностная схема v2.3 — примари-комбинатор + арт — выпилена: она ломала блюпринты, в чертёж попадал только прозрачный комбинатор):
  - **22 прототипа `constant-combinator`** `gofarovich-scl-rail-<rep>` — по классу-орбите масок на прототип; поворот внутри класса = `direction` (N/E/S/W). Контракт «маска ↔ (прототип, direction)» — `scripts/railmask.lua` (общий для data и runtime, 64 маски = 22 орбиты C4 по Бёрнсайду, самопроверка assert'ом).
  - Почему комбинатор: нативно **wire-connectable** (его роль ещё с v2.3; нативный GUI подавляем), нативно **направленный** (`supports_direction` всегда true — без хаков и дроп-стрелок), **хранит direction в блюпринтах** → чертежи видимы и корректно поворачиваются; **не крафт-машина** → нет статуса «работы», сторонние моды-индикаторы (Bottleneck и т.п.) рельсы не видят. **Арт в `integration_patch`** (Sprite4Way, ячейки того же листа rail.png через x/y) на `integration_patch_render_layer = "lower-object"` → лежит на земле, принимает тени зданий, всегда под каретками; `integration_patch` — свойство `EntityWithHealthPrototype`, есть у всех строимых. Собственные `sprites` комбинатора пусты. `simple-entity-with-owner` по-прежнему не подключается к проводам.
  - История отказа от `assembling-machine` (v2.5.x): у машины неотъемлемый статус работы (чужие моды фильтруют по типу) и направленность достигалась только хаками (`vector_to_place_result` рисует неубираемую стрелку; квадратная 1×1-машина без него — `supports_direction=false`).
  - Обвес: `hidden = true`, `placeable_by`/`minable.result` → один item `gofarovich-scl-rail` (ставит «крест» rail-3, скрипт тут же морфит по маске).
  - **Флип чертежей (F/G) НЕ поддержан:** у комбинаторов нет mirroring-бита (в 2.0.76 `use_mirroring` не существует, в 2.1 он только у крафт-машин). Флип хиральных уголков давал бы молча неверную маску. Ваниль решает флип тремя способами, ни один нам не доступен: ленты не хранят кривизну (выводится из соседей); рельсы держат 16-way direction, где зеркало кривой = другое значение direction (пары d↔16−d) — а спрайты комбинатора жёстко Sprite4Way; машины (2.1) — mirror-бит + предотрисованные `graphics_set_flipped`. Runtime при этом mirroring-aware (маска = f(имя, direction, mirroring), D4-ремап сторон в тегах) — если база когда-нибудь станет зеркалируемой, включится само.
  - **Морф** (rails.lua): смена eff_mask → внутри класса пишем `entity.direction`, между классами — пересоздание с переносом проводов (`snapshot_wires`/`restore_wires`). `node.entity` репоинтится, ключ тайла стабилен, открытые GUI рефрешатся хуком `R.on_geometry_changed` → `GUI.refresh_key`.
  - **Ручной поворот (R) по рельсу запрещён по дизайну** — геометрия правится только галочками GUI / авто-соседями. Комбинатор направленный нативно, движок R разрешает — откатываем в `on_player_rotated_entity` (`direction = previous_direction`); страховочный откат флипа (`on_player_flipped_entity`) тоже стоит. Поворот чертежа — пожалуйста, там всё ремапится.
- **Каретка** = `simple-entity-with-owner` (НЕ `car`/`unit`), 1×1, нулевая коллизия. Реальный entity → родной свет/тень.
  - `flags` включают **`placeable-off-grid`** — иначе `teleport` снаппит на центр тайла. Критично.
  - Позиция: только `teleport`, дробные координаты. `speed` **не задавать никогда** (ноль физики).
  - Поворот: через `graphics_variation` (**не** `direction`). `pictures` = лист 16 кадров, `random_variation_on_create = false`; в runtime `graphics_variation = facing` (1..16).
  - Груз (M7): число слотов = **порядковый номер качества каретки** — normal 1, uncommon 2, rare 3, epic 4, legendary 5 (`G.slots_by_quality(entity)` — идём по цепочке качеств от `normal`, считая позицию; **не** `1 + quality.level`: движок держит дыру на уровне 4 под модовые тиры, `legendary.level = 5` → `1+level` дал бы 6 слотов; у simple-entity своего инвентаря нет — размер скриптового считаем сами). Размер выставляется **сразу в `cart_register`** по качеству; клон/блюпринт качество несут движком → новый инвентарь того же размера (клон копирует и содержимое послотово, блюпринт — только качество). Старые фикс-4 инвентари мигрируют в `rebuild_world` (`C.fit_cart_inventory`): resize под качество, усадка (4→1 у normal) спиллит лишний груз на поверхность. Живёт в `storage.carts[un].inv` (LuaInventory сериализуем), создаётся **сразу в `cart_register`** (груз — суть каретки, как у вагона/сундука; ленивое досоздание в `cart_inventory` — только миграция старых записей). Окно — **нативное** (`player.opened = inv`, поддержано в 2.0), открытие — штатная «открыть» (E) через linked custom-input `open-gui` (`gofarovich-scl-open-cart`). **Звуки** открытия/закрытия — скриптом (`play_sound`, SoundPath `entity-open/close` железного сундука; у окна script-инвентаря своих нет; факт открытия — `storage.cart_open`). Вьюпорт с кареткой над слотами пробовали и **убрали**: relative-GUI встаёт только вокруг ВСЕГО окна (top/bottom/left/right), не между титулом и слотами, а внутрь нативного окна мод-элементы не добавляются. Добыча каретки возвращает груз добытчику (`event.buffer`), смерть/блэкаут — груз гибнет, клон копирует послотово. Id: `unit_number`.
- Стиль: data-oriented, главный цикл наш (`on_tick`). Всё детерминированно (мультиплеер): фиксированный порядок обхода и тай-брейки.
- **Инвариант:** каждая каретка всегда принадлежит ровно одному составу (размер ≥ 1). Соло-каретка = состав из одного, тот же код-путь.
